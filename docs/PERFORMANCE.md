# 性能与容量

## 选择后端

原生后端复用同一 SQLite 连接，没有进程启动和 JSON 输出开销，是生产默认。CLI 每次请求启动 `sqlite3`，只适合作为兼容和排障路径。

高频重复 SQL 应复用 `SQLite原生预编译`。多条写入使用显式立即事务或 `批量执行`，避免每条语句独立提交。WAL 适合读写并发；同步等级应依据持久性要求选择，不要为基准盲目关闭同步。

## 固定上限

| 项目 | 上限 |
| --- | --- |
| SQL | 1 MiB |
| 参数 | 65,536 |
| 查询行 | 100,000 |
| 查询列 | 1,024 |
| 查询结果 | 16 MiB |
| JSON 路径段 | 64 |
| JSON 路径 | 4,096 字符 |
| 单段迁移 SQL | 1,048,576 字符 |
| 迁移数 | 10,000 |

SQLite 编译时自身限制可能更低，驱动信息会报告实际最大变量数。

## 基准方法

发布比较应固定同一机器、SQLite 制品、日志/同步模式和数据库形态，分别测量：

- 预编译单行插入；
- 单事务批量插入；
- 有索引与无索引参数查询；
- 表结构反射；
- 规范言据文本与 JSON 原生结构转换；
- 迁移规划与提交。

报告必须列出预热、轮数、数据规模、中位数和高分位，不把 CLI 与原生结果混为同一后端。本仓库目前不发布跨机器性能数字，避免把未校准结果写成稳定承诺。
